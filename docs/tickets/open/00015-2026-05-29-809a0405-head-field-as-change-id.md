# Head field stores bare `change_id`, not `trunk()..<id>`

## Goal
Switch the Generate-screen head form field to store and display the bare change_id of the selected revset's tip. The `trunk()..<change_id>` form remains the internal revset *query string* used for jj invocations, but the user-visible form field, picker options, and validation see only the change_id.

## Context
`RevsetSummary::label` is constructed as `trunk()..<change_id>` (see `src/jj.rs:146`). That same label is stuffed into `form.head` as the default and via `sync_head_from_selected_revset` (`src/generate.rs:1123, 1267`). The user reported that this leaks an internal revset expression into a form field that should be a change_id. It also defeats the `validate_for_execution` guard at `src/generate.rs:1604-1609`, which checks `is_change_id_like(head) && base == head` â€” that check currently never fires because the head is the label, not a change_id. Downstream `jj bookmark create/move <name> <revision>` calls (`src/app.rs:1141`, `src/generate.rs:1659-1661`) expect a single revision, so passing the bare change_id is correctness-improving.

## Non-Goals
- Do not change the revset query string used to call jj â€” `trunk()..<change_id>` is still the right argument to `jj log -r <revset>` and friends.
- Do not change the Menu pane's row labels â€” those can keep showing the human-readable revset description; only the form field needs the bare change_id.
- Do not rework picker selection semantics, only the value/label of each option.
- Do not change validation rules; only fix the input they receive.

## Design Decisions
- Add `RevsetSummary::tip_change_id(&self) -> Option<&str>` returning `self.change_ids.first().map(String::as_str)`.
- Wherever the head field currently receives the revset label, use `tip_change_id().unwrap_or("")` instead.
- `head_picker_options` (`src/generate.rs:1505`): each option's value is the change_id; the option's display can include the revset description for context, but the *value* committed to the field must be the bare change_id.
- `validate_for_execution` (`src/generate.rs:1604`): unchanged in shape, but the `is_change_id_like(head)` guard now actually fires.
- `tip_bookmark` derivation (`src/generate.rs:1637`): inspect; if it expects the head string to be a change_id, this fix is what it always wanted.
- Default branch logic (`src/generate.rs:1127-1130`) stays as-is.
- The Menu pane row label (left column) is unchanged â€” still shows the existing description and `trunk()..<id>` revset summary text in the secondary line. Only the head form field changes.

## Implementation Plan
1. In `src/generate.rs`, add `pub fn tip_change_id(&self) -> Option<&str>` on `RevsetSummary`.
2. Update `GenerateState::new` (around `src/generate.rs:1123-1132`) so `default_head` uses `tip_change_id`.
3. Update `sync_head_from_selected_revset` (around `src/generate.rs:1267-1282`) so the value written to `form.head` is `selected_revset().tip_change_id().unwrap_or("").to_string()`.
4. If ticket `enter-force-syncs-head` has landed first, also update `force_sync_head_from_selected_revset` the same way.
5. Update `head_picker_options` (around `src/generate.rs:1505`) so each `PickerOption` uses the change_id as its value. The option label may include the description for readability but the value must be the change_id.
6. Update tests that assert the old label form:
   - `src/generate.rs:1787` (revset constructor in tests; verify head expectations).
   - `src/generate.rs:2087` (`head is required` test path).
   - `src/generate.rs:2133` (CLI args including `--head`).
   - `src/generate.rs:2296` (`base==head when both change_ids` test â€” should now actually exercise the validation).
7. Verify `tests/windows_pr_generation_integration.rs` still passes; update any assertions that compared head to the label form.
8. Smoke `jj bookmark create/move` arguments in `src/app.rs:1141` and `src/generate.rs:1659-1661` to confirm they now receive a single revision.
