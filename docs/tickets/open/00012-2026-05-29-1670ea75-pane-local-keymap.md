# Pane-local key dispatch on Generate screen

## Goal
Restructure the Generate-screen normal-mode key handling so each pane (Menu, Form, Preview) owns its action keys. Pane-specific shortcuts (`g`, `c`, `p`, `i`, `r`) must no-op when their owning pane is not focused. Truly global keys (`Tab`, `Shift+Tab`, `Esc`, `q`/`Ctrl+C`) keep working everywhere.

## Context
Today `apply_normal_key` in `src/app.rs` maps keys globally on the Generate screen (around `src/app.rs:558-566`). The user reports the keymap feels broken because the same key has different meanings in different panes and the footer hints do not reflect where each key actually fires. Making each pane a small modal scope is the first foundational step for the subsequent footer hint, force-sync, head-field, blocker banner, and spinner tickets. Background: see `docs/design.md` for the three-pane Generate layout.

## Non-Goals
- Rewriting the footer hint line √¢‚Ç¨‚Äù that lives in a separate ticket (`pane-aware-footer-hints`).
- Changing what `g` / `c` / `p` actually do once they fire. Only the dispatch gating changes here.
- Touching `InputMode::Editing` / `InputMode::Confirm` handling √¢‚Ç¨‚Äù those already have dedicated arms.
- Adding new actions or new focus targets.

## Design Decisions
- Pane-local map for Generate / Normal:
  - Menu: arrows / `j` / `k` move selection, `Enter` picks the revset (existing `select_revset`), `r` refreshes revsets.
  - Form: arrows / `j` / `k` move field, `Enter` and `i` begin edit, `g` runs `generate_pr`.
  - Preview: arrows / `j` / `k` scroll, `p` toggles prompt view, `g` regenerates, `c` confirms execution.
- Truly global on Generate / Normal: `Tab`, `Shift+Tab`, `Esc`, `q`, `Ctrl+C`.
- Implementation shape: keep a single `apply_normal_key` entry point but match on `(self.screen, self.focus, key.code)`. Out-of-pane keys return `Action::Tick` (current pattern for unhandled keys) so they are silently ignored.
- `generate_pr` and `confirm_execution` keep their existing screen/phase guards; we are only changing the *dispatch* layer, not the handlers' internal checks.
- Do not introduce a new enum for pane-modes; the existing `Focus` enum is the mode.

## Implementation Plan
1. In `src/app.rs`, restructure the Generate-screen normal-mode branch of `apply_normal_key` (around `src/app.rs:558-566`):
   - Keep the global pre-checks (Esc/Tab/Shift+Tab/q/Ctrl+C) above the pane-local match.
   - Add a `match self.focus` switch that produces the correct `Action` for `g`, `c`, `p`, `i`, `r`, and `Enter`.
   - Arrow / `j` / `k` already route through `Direction::Up`/`Down` √¢‚ÄÝ‚Äô `navigate` which is focus-aware; keep that.
2. Audit other Generate-screen handlers (`Action::Generate`, `Action::Confirm`, `Action::TogglePromptView`, `Action::Refresh`) to make sure no callers assume cross-pane firing. Add a focus guard inside each handler only if dispatch alone is not enough (prefer dispatch).
3. Update the existing app-level tests in `src/app.rs` that send `g` / `c` / `p` / `i` / `r` keys without setting `focus`. Set `app.focus` explicitly to the owning pane before pressing. Affected tests include around lines 1717, 1756, 1837, 1904, 2121, 2161, 2175, 2208 (these may have shifted; search for `KeyCode::Char('g')`, `KeyCode::Char('c')`, and `KeyCode::Char('p')`).
4. Add one new focused test: pressing `p` while `focus == Focus::Menu` results in `Action::Tick` (or equivalent no-op) and does not toggle `prompt_view`.

## Acceptance Criteria
- On the Generate screen in `InputMode::Normal`:
  - `g` only triggers `Action::Generate` when `focus == Focus::Form` or `focus == Focus::Preview`.
  - `c` only triggers confirm when `focus == Focus::Preview`.
  - `p` only toggles prompt view when `focus == Focus::Preview`.
  - `i` only begins editing when `focus == Focus::Form`.
  - `r` only refreshes when `focus == Focus::Menu`.
  - `Enter` picks revset on Menu and begins edit on Form (existing behavior, now pane-local).
- Tab/Shift+Tab/Esc/q/Ctrl+C continue to work from all three Generate panes.
- All existing tests pass after focus has been set explicitly where required.
- New test confirms `p` is a no-op outside Preview focus.

## Risks
- Existing tests rely on focus state that was implicit; missing one update will surface as a flipped action assertion.
- Macro-style `match` over `(screen, focus, code)` can become unwieldy √¢‚Ç¨‚Äù keep it readable by extracting a small `fn dispatch_generate_normal(...)` helper if the match grows.
- `Enter` already does two things (pick revset on Menu, begin edit on Form). Make sure the dispatch keeps both, not just one.
